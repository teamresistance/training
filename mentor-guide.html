Team Resistance #86 - Programming Mentor Quick Guide

Supporting New Team Members (2025 Season)

---

Overview

This guide helps experienced programmers effectively mentor beginners through the challenge progression. Each beginner should be paired with one experienced programmer throughout their training.

---

Mentoring Philosophy

The "I Do, We Do, You Do" Approach

1. I Do: Demonstrate the concept/technique 2. We Do: Work through it together with the beginner 3. You Do: Let them try independently while you observe

Key Principles

  • Be Patient: Remember when you were learning!
  • Ask Questions First: "What do you think we should do?" before giving answers
  • Encourage Experimentation: Let them make mistakes in a safe environment
  • Celebrate Wins: Acknowledge progress, even small victories
  • Connect to Real Robot: Relate challenges to actual competition robots

Challenge-by-Challenge Mentor Notes

Challenge 1: Simple Motor Control

Common Struggles:
  • Understanding CAN IDs and device initialization
  • Controller button mapping confusion
  • Not understanding the loop structure (iterativeInit vs. iterativePeriodic)
Key Teaching Points:
  • Explain WHY we use negative values for reverse (it's just a convention)
  • Show them how to find SparkMAX documentation
  • Emphasize safety: "Always think about what happens if something goes wrong"
Mentor Tips:
  • Have them explain each line back to you
  • Show them how to use the Driver Station to test buttons
  • Demonstrate on the actual robot first, then have them try
Red Flags:
  • Copying code without understanding
  • Not testing incrementally
  • Ignoring safety limits

---

Challenge 2: Sensor Feedback & SmartDashboard

Common Struggles:
  • NetworkTables and SmartDashboard relationship confusion
  • Understanding encoder units (ticks vs rotations vs degrees)
  • Figuring out update rates and periodic functions
Key Teaching Points:
  • Draw a diagram showing Robot â†’ NetworkTables â†’ SmartDashboard â†’ Driver Station
  • Explain sensor fusion concepts at a high level
  • Show them how to use Glass and AdvantageScope as alternatives
Mentor Tips:
  • Have them predict what values they'll see before running code
  • Teach them to use console logging for debugging
  • Show how to organize SmartDashboard layouts
What to Demo:
  • Manually rotating the motor and watching encoder values
  • The relationship between speed and position
  • How to spot encoder issues (disconnected, wrong direction, etc.)
Red Flags:
  • Not checking if values make sense (e.g., motor at 10000 RPM)
  • Updating SmartDashboard too frequently (performance issues)
  • Not labeling values clearly

Challenge 3: PID Position Control

NOTE: This is where many beginners struggle. Be extra available. Pre-Challenge Prep:
  • Review PID theory together (recommend watching a video first)
  • Have them use the team's PIDlab tool to experiment
  • Discuss real-world examples (cruise control, thermostat)
Common Struggles:
  • Not understanding what each term (P, I, D) does
  • Random tuning without methodology
  • Integral windup
  • Derivative kick
  • Confusion about error sign
Key Teaching Points:
  • Start with P only: Set I and D to zero, tune P first
  • Explain each term with an analogy:

- P: "How hard you push based on how far you are" - I: "Making up for persistent errors over time" - D: "Slowing down as you approach target"

  • Show them oscillation, overshoot, undershoot
  • Teach systematic tuning: P â†’ D â†’ I (not I!)
Mentor Tips:
  • Have them graph the response in NetworkTables/AdvantageScope
  • Start with small movements (1-2 rotations)
  • Use the PIDlab simulation extensively before hardware
  • Teach them to recognize oscillation patterns
What to Demo:
  • Show an over-tuned system (oscillates)
  • Show an under-tuned system (slow/sluggish)
  • Show a well-tuned system
  • Demonstrate what happens without limits (dangerous!)
Debugging Help:
  • Motor doesn't move â†’ Check P gain (probably too small)
  • Motor oscillates â†’ P too high or D too low
  • Steady-state error â†’ Add small I term
  • Motor "hunts" around setpoint â†’ Reduce P or increase D
  • Motor goes full speed indefinitely â†’ Check setpoint and error sign
Red Flags:
  • Using huge P values (>1.0) without understanding
  • Enabling I term before P and D are tuned
  • Not implementing safety limits
  • Tuning on hardware without simulation first
Advanced Discussion Topics:
  • Why use PID instead of bang-bang control?
  • Trade-offs: speed vs smoothness vs accuracy
  • When is PID not the right solution?

Challenge 4: Subsystem Architecture

Common Struggles:
  • Understanding the command-based paradigm
  • When to use requires() vs when not to
  • Command scheduling lifecycle
  • Default commands vs button commands
Key Teaching Points:
  • Draw the architecture diagram: Robot â†’ Subsystem â†’ Commands
  • Explain inversion of control: "Commands call subsystem methods"
  • Walk through command lifecycle: initialize â†’ execute â†’ end â†’ isFinished
  • Discuss why this architecture scales better than monolithic code
Mentor Tips:
  • Have them trace execution flow on paper
  • Show them actual team competition code structure
  • Emphasize: Subsystems own hardware, Commands own behavior
  • Review team coding standards for file organization
What to Demo:
  • Show how button bindings work in RobotContainer
  • Demonstrate command interruption
  • Show debug output for command scheduling
  • Tour the team's real subsystem code
Red Flags:
  • Putting business logic in subsystems instead of commands
  • Creating dependencies between subsystems
  • Not using default commands appropriately
  • Overcomplicated command structures

---

Challenge 5: Autonomous Routine

Common Struggles:
  • Thinking sequentially instead of compositionally
  • Understanding parallel vs sequential vs race groups
  • Timeout logic
  • Handling failures gracefully
Key Teaching Points:
  • Draw flowcharts for autonomous routines
  • Explain that CommandGroups are commands too
  • Teach defensive programming: "What if this doesn't work?"
  • Discuss competition scenarios: "What if alliance partner blocks us?"
Mentor Tips:
  • Start with simple 2-step sequences before complex groups
  • Have them watch competition match videos
  • Practice estimating times (how long DOES it take to move 10 feet?)
  • Use simulation to rapidly iterate
What to Demo:
  • Show competition auto routines from previous years
  • Demonstrate auto selector and how teams choose strategies
  • Show what happens when auto fails mid-match
Advanced Topics:
  • Path planning basics
  • Using triggers and events in PathPlanner
  • Telemetry during auto for debugging
Red Flags:
  • No timeout safety
  • Hardcoded delays instead of sensor feedback
  • Not testing failure modes
  • Overly complex first attempt

---

Challenge 6: Integration Project

NOTE: This is where students transition to real contributors Common Struggles:
  • Coordinating multiple subsystems safely
  • State machine complexity
  • Race conditions
  • Not thinking through all state transitions
Key Teaching Points:
  • Draw state machine diagrams on whiteboard
  • Discuss real-world examples from team's competition robots
  • Teach defensive interlocks: "What could go wrong?"
  • Emphasize testing: "Test every state transition"
Mentor Tips:
  • This is a capstone - let them design more independently
  • Code review is critical here
  • Have them present their design before implementing
  • Connect to actual mechanisms on team robot
What to Demo:
  • Show a real 2-mechanism coordination from competition code
  • Demonstrate what happens when interlocks aren't implemented
  • Walk through the team's competition robot state machine
Assessment Questions:
  • "What happens if the elevator stalls?"
  • "What if robot loses power mid-movement?"
  • "How do you test this safely?"
  • "Why did you choose this state machine design?"
Red Flags:
  • Not drawing out state machine first
  • Missing state transitions
  • No safety interlocks
  • Can't explain their own code

General Mentoring Strategies

Teaching Code Review

As you work through challenges, teach them to review their own code:

The "CLEAN" Checklist:
  • Compiles without warnings
  • Logical structure (good variable names, comments)
  • Error handling (safety, timeouts, nulls)
  • Architecture follows patterns (subsystems, commands)
  • No magic numbers (use named constants)

Handling Frustration

When a beginner gets stuck or frustrated:

1. Take a break: Step away for 5 minutes 2. Rubber duck it: Have them explain the problem out loud 3. Binary search: Comment out half the code to isolate the issue 4. Fresh eyes: Show the code to another mentor 5. Sleep on it: Sometimes tomorrow is better

Encouraging Independence

Early Stage: Provide direct answers when asked Developing Stage: Answer questions with questions: "What do you think?" Intermediate Stage: Point to resources: "Check the WPILib docs for..." Advanced Stage: Let them struggle (productively) before helping

Adjust your approach based on the student's progress through the challenges, not on a fixed timeline.

Encouraging Independence## Build Season Transition

Preparing for Competition Season

Help students understand where they fit in competition season based on their skill level:

Beginner Roles (Levels 1-2):
  • Telemetry and dashboard development
  • Testing and validation
  • Documentation
  • LED/indicator systems
  • Controller button mapping
Intermediate Roles (Levels 2-3):
  • Subsystem implementation
  • Command creation
  • Autonomous strategy implementation
  • Integration testing
Advanced Roles (Levels 3-4):
  • Drive system
  • Vision integration
  • System architecture
  • Code review leadership
  • Mentoring Level 1 programmers

Setting Expectations

  • Competition code has higher quality standards
  • Multiple people editing same codebase
  • Deadlines are real (competition dates fixed!)
  • Not everyone gets to work on "cool" stuff (someone has to do LEDs)
  • Communication becomes critical

When to Escalate

Contact programming leads if:

  • Student shows no progress after multiple sessions
  • Personality conflicts with mentor pairing
  • Student needs different teaching approach
  • Advanced student needs harder challenges
  • Safety concerns with testing approach
